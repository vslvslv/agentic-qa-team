# Contributing Guide

Thanks for improving qa-agentic-team. This guide covers the workflow, versioning rules, and PR requirements.

## Setup

```bash
git clone https://github.com/vslvslv/agentic-qa-team
cd qa-agentic-team
bash bin/dev-setup      # creates ~/.claude/skills/qa-agentic-team → this repo (namespace symlink)
```

Dev mode uses a single namespace symlink so every skill is available as `/qa-agentic-team:<skill-name>`. Edits to `.tmpl` files take effect after regenerating docs — no reinstall needed.

When finished: `bash bin/dev-teardown`

## Editing an existing skill

1. Edit `<skill>/SKILL.md.tmpl` — **never** edit `SKILL.md` directly
2. Regenerate: `npm run gen:skill-docs`
3. Verify freshness: `npm run check:skill-docs`
4. Test your change by running the skill in a sample project
5. Bump version (see below) and add a CHANGELOG entry
6. Open a PR — CI validates docs freshness and version bump

## Versioning

This repo uses 4-part semantic versioning: `MAJOR.MINOR.PATCH.MICRO`

| Bump | When |
|------|------|
| `major` | Breaking change to a skill's interface or output format |
| `minor` | New skill or significant new capability added |
| `patch` | Bug fix, content improvement, or small enhancement |
| `micro` | Typo fix, minor wording, nightly refinement run |

```bash
# Calculate next version
bash bin/qa-team-next-version minor   # prints: 1.15.0.0 → 1.16.0.0

# Update VERSION file
echo "1.16.0.0" > VERSION
```

Every PR that changes skill content **must** bump `VERSION` and add a `CHANGELOG.md` entry. The `version-gate` CI job blocks merges that don't.

## CHANGELOG format

```markdown
## v1.16.0.0 — 2026-05-10 — short summary

### <skill-name>
- What changed and why
```

## CI gates

Two CI jobs run on every PR:

- **`version-gate`**: checks that `VERSION` was bumped and `CHANGELOG.md` has a new entry
- **`skill-docs`**: runs `npm run check:skill-docs` — fails if any `SKILL.md` is stale vs its `.tmpl`

Both must pass before merge.

## Adding a new skill

See [Adding a New Skill](adding-a-skill.md) for the complete step-by-step checklist.

## Code style

- SKILL.md.tmpl files use Markdown with fenced bash blocks
- Preamble bash: use `_VAR` naming for internal variables; always print `KEY: VALUE` lines
- Phases are numbered (Phase 1, Phase 2, etc.) with descriptive names
- Every skill emits a CTRF JSON file to `$_TMP/qa-<name>-ctrf.json`
- Agent files (`.claude/agents/*.md`) mirror the SKILL.md.tmpl body, condensed, without the version check section

## What doesn't need a PR

- Updating reference guides in `*/references/` — these are generated by refinement skills and committed directly
- Updating `learning-sources/` catalog files — also generated
- Nightly refinement log entries in `logs/`
