## 1. k6 Studio

k6 Studio is an **open-source desktop app** (macOS, Windows, Linux) that bridges the gap between browser interactions and k6 load test scripts.

### Workflow: HAR → k6 Script

```
Browser interaction → HAR recording → k6 Studio Generator → k6 script → Grafana Cloud
```

**Three components:**

| Component | Purpose |
|-----------|---------|
| **Recorder** | Captures user flows as HAR files |
| **Generator** | Transforms HAR → k6 scripts with rule-based customisation |
| **Validator** | Visually debugs scripts; inspects request/response per step |

### Key use cases

- Non-engineer testers can create load tests without writing JS
- Rapid script prototyping from real user journeys
- Browser scenario recording (navigate, click, assert text)
- Direct upload to **Grafana Cloud k6** and **Synthetic Monitoring**

### Generated script anatomy

Scripts generated by k6 Studio follow standard k6 patterns:

```typescript
import http from 'k6/http';
import { sleep, check } from 'k6';

export const options = {
  vus: 10,
  duration: '30s',
};

export default function () {
  const res = http.get('https://example.com/');
  check(res, { 'status is 200': (r) => r.status === 200 });
  sleep(1);
}
```

After generation, edit the script to parameterise variables, add thresholds, and replace static values with `SharedArray` data.

### Best practices

- **Validate after generation** — use the Validator to inspect actual response bodies before adding checks.
- **Strip irrelevant HAR entries** — remove analytics/CDN requests before generating.
- **Add thresholds** — generated scripts have no thresholds; always add `options.thresholds` before CI use.
- **Parameterise** — replace hardcoded URLs and tokens with `__ENV` variables.

### Core k6 module patterns

```typescript
import { check, group, sleep, fail } from 'k6';
import http from 'k6/http';

export const options = {
  stages: [
    { duration: '1m', target: 50 },   // ramp up
    { duration: '5m', target: 50 },   // steady state
    { duration: '1m', target: 0 },    // ramp down
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],
    http_req_failed: ['rate<0.01'],
  },
};

export default function () {
  group('Product catalog', () => {
    const res = http.get('https://api.example.com/products');

    if (!check(res, { 'status 200': (r) => r.status === 200 })) {
      fail('Products API returned non-200');
    }

    sleep(1);
  });
}
```

### HTTP module — advanced patterns

```typescript
import http from 'k6/http';
import { check } from 'k6';

// Batch parallel requests
export default function () {
  const responses = http.batch([
    ['GET', 'https://api.example.com/users'],
    ['GET', 'https://api.example.com/products'],
    ['GET', 'https://api.example.com/orders'],
  ]);

  responses.forEach((res) => {
    check(res, { 'status 200': (r) => r.status === 200 });
  });
}

// POST with JSON body and auth header
export function postWithAuth(token: string, payload: unknown) {
  const res = http.post(
    'https://api.example.com/orders',
    JSON.stringify(payload),
    {
      headers: {
        'Content-Type': 'application/json',
        Authorization: `Bearer ${token}`,
      },
      tags: { endpoint: 'create-order' },
    }
  );
  return res;
}
```

### Custom metrics

```typescript
import { Counter, Gauge, Rate, Trend } from 'k6/metrics';
import http from 'k6/http';

const checkoutErrors = new Counter('checkout_errors');
const activeUsers = new Gauge('active_users');
const successRate = new Rate('checkout_success_rate');
const checkoutDuration = new Trend('checkout_duration_ms', true); // true = time metric

export default function () {
  activeUsers.add(1);
  const start = Date.now();

  const res = http.post('https://api.example.com/checkout', '{}');

  checkoutDuration.add(Date.now() - start);
  const ok = res.status === 200;
  successRate.add(ok);
  if (!ok) checkoutErrors.add(1);

  activeUsers.add(-1);
}
```

### Browser module — Web Vitals

```typescript
import { browser } from 'k6/browser';
import { check } from 'k6';

export const options = {
  scenarios: {
    browser_test: {
      executor: 'constant-vus',
      vus: 1,
      duration: '30s',
      options: { browser: { type: 'chromium' } },
    },
  },
};

export default async function () {
  const page = await browser.newPage();
  try {
    await page.goto('https://example.com/');
    await page.waitForLoadState('networkidle');

    // Collect Web Vitals via browser metrics
    check(page, {
      'title present': (p) => p.title() !== '',
    });
  } finally {
    await page.close();
  }
}
```

## 3. xk6-disruptor — Kubernetes Fault Injection

xk6-disruptor injects faults (errors + latency) into Kubernetes Pods and Services **from within k6 scripts**, without deploying extra agents or operators.

### When to use

- Validating retry/backoff logic under HTTP 5xx
- Testing circuit breaker thresholds
- Measuring SLO degradation with injected latency
- Combining load testing + chaos in a single run

### Key classes

| Class | Target | Fault types |
|-------|--------|------------|
| `PodDisruptor` | Individual Pods | HTTP errors, gRPC errors, delays |
| `ServiceDisruptor` | Kubernetes Services | HTTP errors, gRPC errors, delays |

### Typical usage pattern

```typescript
// NOTE: xk6-disruptor requires k6 built with the extension
// k6 build --with github.com/grafana/xk6-disruptor
import { PodDisruptor } from 'k6/x/disruptor';

// Disruptor setup runs once (not per VU iteration)
export function setup() {
  const disruptor = new PodDisruptor({
    namespace: 'default',
    select: { labels: { app: 'payment-service' } },
  });

  // Inject 10% HTTP 500 errors + 50ms delay for 1 minute
  disruptor.injectHTTPFaults(
    {
      averageDelay: '50ms',
      errorRate: 0.1,
      errorCode: 500,
    },
    '1m'   // duration
  );
}

export default function () {
  // Normal k6 load test runs while faults are injected
  const res = http.get('https://payment.example.com/health');
  check(res, {
    'resilient under fault': (r) => r.status === 200 || r.status === 503,
  });
}
```

### ServiceDisruptor pattern

```typescript
import { ServiceDisruptor } from 'k6/x/disruptor';

export function setup() {
  const disruptor = new ServiceDisruptor({
    name: 'payment-svc',
    namespace: 'default',
  });

  // Inject gRPC errors
  disruptor.injectGrpcFaults(
    {
      statusCode: 14,         // UNAVAILABLE
      errorRate: 0.05,        // 5% of requests
      averageDelay: '100ms',
    },
    '2m'
  );
}
```

### Best practices

- **Combine with thresholds**: set `options.thresholds` to assert SLO compliance *during* fault injection.
- **Use `setup()` for fault injection** — ensures faults are injected once before VUs start, not per-VU.
- **Clean up**: disruptors auto-remove faults when duration expires; use `teardown()` for explicit cleanup.
- **Start with low error rates** (1–5%) to validate test infrastructure before raising.
- **Namespace isolation**: always specify `namespace` to avoid accidentally targeting production.

## 4. k6 Experimental Modules

> **Warning**: These modules are stable-enough for production use but may have breaking API changes in future k6 releases. Pin k6 versions in CI.

### CSV — efficient file parsing

```typescript
import csv from 'k6/experimental/csv';
import { SharedArray } from 'k6/data';

// Parse CSV once in init context (memory efficient)
const rows = new SharedArray('testdata', async () => {
  const file = await csv.parse(open('./fixtures/users.csv'), {
    delimiter: ',',
    skipFirstLine: true,  // skip header row
  });
  return file;
});

export default function () {
  const row = rows[__VU % rows.length];
  const [email, password] = row;
  // use email, password
}
```

### Filesystem — memory-efficient I/O

```typescript
import { open } from 'k6/experimental/fs';

// Opened once, seekable — avoids loading entire file into memory
const file = await open('./large-payload.json');

export default async function () {
  // Read a chunk at current position
  const buf = new Uint8Array(1024);
  const bytesRead = await file.read(buf);

  // Seek back to start for next VU iteration
  await file.seek(0, 'start');
}
```

### Streams — Readable stream API

```typescript
import { ReadableStream } from 'k6/experimental/streams';

export default async function () {
  const stream = new ReadableStream({
    start(controller) {
      controller.enqueue('chunk-1');
      controller.enqueue('chunk-2');
      controller.close();
    },
  });

  const reader = stream.getReader();
  let result;
  while (!(result = await reader.read()).done) {
    console.log(result.value);
  }
}
```

### When to use experimental modules

| Use case | Module |
|----------|--------|
| Large CSV test data (> 10K rows) | `k6/experimental/csv` |
| Large file payloads (multi-MB request bodies) | `k6/experimental/fs` |
| Streaming response processing | `k6/experimental/streams` |

## 5. k6/secrets — Secure Credential Management

The `k6/secrets` module prevents credential leakage by **auto-redacting secret values** in all console output and logs.

### Why it matters

Without secrets management, patterns like `const token = __ENV.API_TOKEN` expose values if `console.log` is called anywhere in the test. The secrets module ensures values are redacted automatically.

### Core API

```typescript
import secrets from 'k6/secrets';

// Access from the DEFAULT secret source
export default async function () {
  const apiKey = await secrets.get('API_KEY');
  // apiKey holds the real value internally
  console.log(apiKey); // logs: ***SECRET_REDACTED***

  const res = http.get('https://api.example.com/data', {
    headers: { Authorization: `Bearer ${apiKey}` },
  });
}
```

### Named secret sources

```typescript
import secrets from 'k6/secrets';

export default async function () {
  // Access a named source (useful for multi-environment setups)
  const prodKey = await secrets.source('prod-vault').get('DB_PASSWORD');
  const stagingKey = await secrets.source('staging-env').get('DB_PASSWORD');
}
```

### Secret source configuration

Secret sources are configured at runtime via k6 CLI flags or environment:

```bash
# Vault-backed source
k6 run --secret-source vault=url=https://vault.example.com,token=$VAULT_TOKEN script.js

# Environment variable source  
k6 run --secret-source env script.js
```

### Migration from `__ENV`

```typescript
// BEFORE: insecure
const token = __ENV.API_TOKEN;  // leaks in logs

// AFTER: secure
const token = await secrets.get('API_TOKEN');  // auto-redacted
```

### Best practices

- **Always use `k6/secrets` for credentials** — API keys, DB passwords, JWT secrets.
- **Never use `__ENV` for secrets** — `__ENV` values are visible in logs and reports.
- **Use named sources** to separate secret collections by environment (dev/staging/prod).
- **CLI-configure sources** via `--secret-source` flags; avoid hardcoding source config in scripts.
- **Rotate secrets independently** — k6 scripts don't need to be updated when secrets rotate.
