## Core Principles

### 1. Consumer-Driven Contracts: The Consumer Defines What It Needs
In consumer-driven contract (CDC) testing, each **consumer** service specifies exactly which parts of a **provider's** API it depends on — not the full API spec. The provider then verifies it can satisfy every consumer's contract. This inverts the traditional "provider publishes a spec, consumers conform" model. The WHY: providers learn which fields are actually used, so they can safely evolve unused parts without breaking anyone.

### 2. The Pact as a Living Document
A *pact* is a JSON file that encodes: (a) the request the consumer will make, (b) the minimum response shape the consumer needs. It is generated by running consumer-side test cases against a mock provider (Pact's mock service), not against the real network. The WHY: the consumer test suite becomes the single source of truth for what the integration must support.

### 3. Contract Testing Sits Between Unit and Integration Tests
Contract tests verify the *shape* of the API boundary — request/response structure, status codes, required fields. They do NOT verify business logic ("does the order service calculate tax correctly?"). That belongs in unit tests. End-to-end (E2E) environment wiring belongs in integration tests. CDC fills the gap: it catches breaking API changes cheaply, without spinning up full stacks.

### 4. The Pact Broker as the Shared Contract Store
The Pact Broker (open-source or PactFlow SaaS) stores published pact files, tracks which provider versions have verified which consumer versions, and provides a "can I deploy?" API. Without a broker, teams share pact JSON via repos or CI artifacts — this works but loses traceability. The WHY: the broker decouples consumer CI from provider CI and enables safe independent deployment.

### 5. Shift Integration Verification Left, Not Right
Conventional integration tests require both services to be running simultaneously — they're slow, expensive, and flaky. CDC testing decomposes the verification: consumers test against a local mock; providers replay recorded interactions against their real code. Neither side needs the other deployed. The WHY: fast feedback, no environment dependency, and clear ownership of each side of the contract.

## When to Use

- **Microservices with REST or GraphQL boundaries** — the primary sweet spot. Each service team can iterate independently and trust the contract to catch regressions.
- **Internal API versioning** — when two teams (e.g., mobile app team + backend team) co-evolve an API and want fast feedback without a shared staging environment.
- **High-traffic APIs consumed by multiple clients** — where a single provider change could break many consumers. The Pact Broker "can-i-deploy?" check prevents merging breaking changes.
- **Greenfield services** — ideal from day one; retrofitting is painful but still worth it for critical integrations.

**Do NOT use contract testing when:**
- You own both sides of the integration and they are always deployed together (a monolith or monorepo deployed as a unit).
- The API is a third-party external service you cannot influence — use a recorded stub/VCR cassette or Wiremock instead.
- The integration is event-driven via message queues — use a message contract tool (e.g., Pact's message pact support) rather than HTTP contract tests; mixing them causes confusion.
- Your team is smaller than ~5 engineers and the overhead of maintaining Pact Broker outweighs the value.

## Patterns

### Pattern 1: Consumer Test with Pact Mock Server

The consumer test defines an *interaction*: "when I call `GET /users/1`, I expect a `200` with at least `{ id: 1, name: string }`." Pact's mock server runs locally and replays this interaction during the test. The generated pact JSON is then published.

```typescript
// consumer/src/user-client.test.ts
import { PactV3, MatchersV3 } from '@pact-foundation/pact';
import { UserClient } from './user-client';
import path from 'path';

const { like, integer, string } = MatchersV3;

const provider = new PactV3({
  consumer: 'FrontendApp',
  provider: 'UserService',
  dir: path.resolve(process.cwd(), 'pacts'),
  logLevel: 'warn',
});

describe('UserClient', () => {
  describe('getUser()', () => {
    it('returns a user when the user exists', async () => {
      // Arrange: define what the consumer will send and what it expects back
      await provider
        .given('user 1 exists')
        .uponReceiving('a request for user 1')
        .withRequest({
          method: 'GET',
          path: '/users/1',
          headers: { Accept: 'application/json' },
        })
        .willRespondWith({
          status: 200,
          headers: { 'Content-Type': 'application/json' },
          body: {
            // Use matchers, not literal values — provider data changes,
            // but the shape and types must stay stable.
            id: integer(1),
            name: string('Alice'),
            email: string('alice@example.com'),
          },
        })
        .executeTest(async (mockServer) => {
          // Act: point the real client at the Pact mock, not the real service
          const client = new UserClient(mockServer.url);
          const user = await client.getUser(1);

          // Assert: validate the consumer-side mapping logic
          expect(user.id).toBe(1);
          expect(user.name).toBe('Alice');
        });
    });

    it('throws a NotFoundError when user does not exist', async () => {
      await provider
        .given('user 999 does not exist')
        .uponReceiving('a request for non-existent user 999')
        .withRequest({ method: 'GET', path: '/users/999' })
        .willRespondWith({ status: 404, body: { error: string('User not found') } })
        .executeTest(async (mockServer) => {
          const client = new UserClient(mockServer.url);
          await expect(client.getUser(999)).rejects.toThrow('User not found');
        });
    });
  });
});
```

Key points: matchers (`like`, `integer`, `string`, `eachLike`) are critical — they check type/shape, not exact values. Using literal values in the expected body couples consumer tests to provider data and causes false failures.

### Pattern 2: Provider Verification

The provider reads published pact files (from disk or Pact Broker) and replays each interaction against the running provider service. Provider states ("user 1 exists") are set up via state handlers before each interaction.

```typescript
// provider/src/user-service.pact.test.ts
import { Verifier, VerifierOptions } from '@pact-foundation/pact';
import { startServer, stopServer } from './test-helpers';
import path from 'path';

describe('UserService — Pact Provider Verification', () => {
  let serverUrl: string;

  beforeAll(async () => {
    serverUrl = await startServer(); // spin up real provider on a dynamic port
  });

  afterAll(async () => {
    await stopServer();
  });

  it('satisfies all consumer pacts', async () => {
    const opts: VerifierOptions = {
      provider: 'UserService',
      providerBaseUrl: serverUrl,

      // Read pacts from local dir (CI typically reads from Pact Broker instead)
      pactUrls: [path.resolve(__dirname, '../../consumer/pacts/FrontendApp-UserService.json')],

      // OR: pull from Pact Broker
      // pactBrokerUrl: process.env.PACT_BROKER_URL,
      // consumerVersionSelectors: [{ mainBranch: true }, { deployedOrReleased: true }],
      // publishVerificationResult: true,
      // providerVersion: process.env.GIT_SHA,

      // State handlers: set up DB/fixtures for each provider state
      stateHandlers: {
        'user 1 exists': async () => {
          await db.users.seed({ id: 1, name: 'Alice', email: 'alice@example.com' });
        },
        'user 999 does not exist': async () => {
          await db.users.deleteWhere({ id: 999 });
        },
      },
      logLevel: 'warn',
    };

    await new Verifier(opts).verifyProvider();
  });
});
```

### Pattern 3: Pact Broker Integration — Can-I-Deploy? Gate

The "can-i-deploy?" check is the linchpin of safe independent deployment. Before merging or deploying, CI queries the Pact Broker to confirm the version being deployed is compatible with all deployed consumer/provider versions.

```typescript
// scripts/can-i-deploy.ts — run in CI before pushing to deploy environment
import { execSync } from 'child_process';

const SERVICE_NAME = process.env.SERVICE_NAME!;
const VERSION = process.env.GIT_SHA!;
const ENVIRONMENT = process.env.DEPLOY_ENV ?? 'production';
const BROKER_URL = process.env.PACT_BROKER_URL!;

function canIDeploy(): void {
  try {
    // Pact CLI: exits non-zero if deployment would break a consumer or provider
    execSync(
      `npx pact-broker can-i-deploy \
        --pacticipant "${SERVICE_NAME}" \
        --version "${VERSION}" \
        --to-environment "${ENVIRONMENT}" \
        --broker-base-url "${BROKER_URL}"`,
      { stdio: 'inherit' }
    );
    console.log(`can-i-deploy: ${SERVICE_NAME}@${VERSION} → ${ENVIRONMENT} — SAFE`);
  } catch {
    console.error(`can-i-deploy: BLOCKED — contract verification failed`);
    process.exit(1);
  }
}

canIDeploy();
```

### Pattern 4: Message Contract Testing (Event-Driven APIs)

HTTP contracts cover synchronous calls. For async message-passing (Kafka, SQS, RabbitMQ), Pact supports *message pacts*. The consumer defines the message payload shape it expects; the provider verifies it can produce that shape.

```typescript
// consumer/src/order-event-handler.pact.test.ts
import { PactV3, MatchersV3, SpecificationVersion } from '@pact-foundation/pact';
import { processOrderCreatedEvent } from './order-event-handler';

const { like, string, integer } = MatchersV3;

const messagePact = new PactV3({
  consumer: 'NotificationService',
  provider: 'OrderService',
  spec: SpecificationVersion.SPECIFICATION_VERSION_V4,
  dir: 'pacts',
});

describe('OrderCreated message contract', () => {
  it('handles the OrderCreated event correctly', async () => {
    await messagePact
      .expectsToReceive('an OrderCreated event')
      .withContent({
        orderId: string('ord_123'),
        userId: string('usr_456'),
        totalAmount: integer(9900),
        currency: string('USD'),
        lineItems: like([{ sku: string('SKU-001'), qty: integer(2) }]),
      })
      .withMetadata({ contentType: 'application/json' })
      .executeTest(async (message) => {
        // The consumer handler must successfully process this message shape
        const result = await processOrderCreatedEvent(JSON.parse(message.contents as string));
        expect(result.notificationSent).toBe(true);
      });
  });
});
```

### Pattern 5: Array and Nested Object Matchers (`eachLike`, `atLeastOneLike`)

When a response contains arrays of objects, use `eachLike` to assert that every element has the correct shape — not that exactly N elements are present. This is a common pitfall that causes consumer test cases to be coupled to provider data volume.

```typescript
// consumer/src/products-client.test.ts
import { PactV3, MatchersV3 } from '@pact-foundation/pact';
import { ProductsClient } from './products-client';
import path from 'path';

const { eachLike, like, integer, string, decimal, boolean: matchBoolean } = MatchersV3;

const provider = new PactV3({
  consumer: 'CheckoutApp',
  provider: 'ProductCatalogService',
  dir: path.resolve(process.cwd(), 'pacts'),
  logLevel: 'warn',
});

describe('ProductsClient', () => {
  it('returns a paginated list of active products', async () => {
    await provider
      .given('at least one active product exists')
      .uponReceiving('a request for active products page 1')
      .withRequest({
        method: 'GET',
        path: '/products',
        query: { page: '1', status: 'active' },
        headers: { Accept: 'application/json' },
      })
      .willRespondWith({
        status: 200,
        headers: { 'Content-Type': 'application/json' },
        body: {
          // eachLike: every element must match the shape — count is flexible
          items: eachLike({
            id: string('prod_abc'),
            name: string('Widget Pro'),
            // Nested objects use like() to match shape
            price: like({
              amount: decimal(9.99),
              currency: string('USD'),
            }),
            inStock: matchBoolean(true),
            // Nested array: eachLike within eachLike for deeply nested shapes
            tags: eachLike(string('electronics')),
          }),
          // Top-level pagination metadata
          page: integer(1),
          totalPages: integer(5),
          totalItems: integer(47),
        },
      })
      .executeTest(async (mockServer) => {
        const client = new ProductsClient(mockServer.url);
        const result = await client.listActiveProducts({ page: 1 });

        // Assert consumer-side mapping: the client correctly maps the response
        expect(result.items).toHaveLength(1); // Pact mock returns exactly 1 per eachLike
        expect(result.items[0]).toMatchObject({
          id: expect.any(String),
          name: expect.any(String),
          price: { amount: expect.any(Number), currency: expect.any(String) },
          inStock: expect.any(Boolean),
        });
        expect(result.page).toBe(1);
        expect(result.totalItems).toBeGreaterThan(0);
      });
  });
});
```

Key: `eachLike(template)` generates a Pact matcher that asserts the array is non-empty and each element matches `template`. The mock server returns one element (the template); this is by design — consumer test cases must not assert exact array length, only shape.

## Anti-Patterns

### Anti-Pattern 1: Testing Too Much in a Contract
Using exact value matchers (`{ name: 'Alice', id: 1 }`) instead of type/shape matchers (`{ name: string(), id: integer() }`) couples the contract to test data. The provider's test database uses different values, causing spurious failures. **WHY it's harmful:** it defeats the purpose of CDC — the consumer now dictates provider data, creating a maintenance burden for every data change.

### Anti-Pattern 2: Contract as Integration Test
Including business logic assertions in contracts ("the response must include a discount field when the user has a premium subscription") turns a shape-verification tool into a functional test. **WHY it's harmful:** contracts become fragile, difficult to verify, and require both services to be tightly synchronized. Business logic belongs in the provider's own unit and integration test suites.

### Anti-Pattern 3: Skipping Provider States
Writing contracts without provider states ("given X exists") forces the provider to maintain a fixed database seed that matches every consumer interaction. **WHY it's harmful:** interactions become order-dependent, the provider seed grows over time, and adding a new consumer interaction risks breaking others. State handlers isolate setup per interaction.

### Anti-Pattern 4: Pact as the Only Integration Safety Net
Treating a passing "can-i-deploy?" check as a substitute for any integration or smoke testing in a staging environment. **WHY it's harmful:** contracts only verify API shape, not runtime behaviour, performance, auth integration, or database schema migrations. A contract-passing deploy can still fail in production.

### Anti-Pattern 5: Committing Pact JSON to the Provider Repo
Storing generated pact files in the provider repository rather than publishing them to a Pact Broker. **WHY it's harmful:** providers must fetch from consumers, creating a coupling between repos. The Pact Broker is purpose-built for version tracking, webhook triggers, and the can-i-deploy matrix.

## Real-World Gotchas  [community]

1. **Provider states become the biggest maintenance burden.** [community] As consumers grow, each adds new states that the provider must implement in state handlers. Teams that start without a state-handler convention (naming, cleanup, isolation) find state drift causes flaky provider verifications within months. WHY: state handlers run in the same DB as other tests unless explicitly isolated; state left by one interaction bleeds into the next.

2. **Pact Broker "can-i-deploy?" checks slow CI if not cached.** [community] The broker query is a network call that resolves against the full compatibility matrix of all registered consumers. Large organisations with 50+ services report 20–40 second waits per service per CI run. WHY: the broker recomputes the matrix on every query without caching. Fix: use `--to-environment` scoping and limit selectors to `mainBranch + deployedOrReleased`.

3. **Consumer teams forget to publish updated pacts after client code changes.** [community] A consumer changes how it calls an endpoint but doesn't re-run the Pact test suite; the pact file in the broker remains stale. The provider verifies the old pact, "can-i-deploy?" passes, but production breaks. WHY: pact generation is a test artefact, not part of the build artefact pipeline. Fix: make pact publication a required CI step gated on test execution, not an optional post-build hook.

4. **Using wildcard `.*` regex matchers instead of typed matchers causes false positives.** [community] Some teams use regex matchers for flexibility ("any string is fine") which lets structurally wrong responses pass verification. WHY: Pact's matcher system is designed to validate type safety, not regex patterns. Loose regex matchers defeat the contract's purpose and miss breaking changes like a field changing from `string` to `null`.

5. **Mixing Pact spec versions (V2 and V4) within the same ecosystem.** [community] Teams that start with Pact V2 and later introduce V4 interactions (for message pacts or plugin support) find the broker stores incompatible pact schemas for the same consumer-provider pair. WHY: V2 and V4 pact files use different JSON schemas; the broker does not transparently merge them, leading to confusing verification failures. Fix: standardise on Pact V4 from the start; V4 is backward compatible for HTTP interactions.

6. **Provider verification runs as a single slow test suite.** [community] All consumer pacts for a provider are verified in one Verifier call. As the number of consumers grows, a single slow state handler makes all subsequent interactions wait. WHY: Pact's default verifier is synchronous per interaction. Fix: run provider verification in parallel jobs per consumer using multiple `pactUrls` entries or by using `PactBrokerWithDynamicConsumers` with consumer-level parallelism.

7. **Contract tests provide false confidence for auth and middleware.** [community] Contract tests typically run without real auth headers (test API keys or bypassed auth). Teams discover in production that their auth middleware rejects requests the contract test never saw. WHY: the provider `startServer()` in tests often uses a permissive middleware stack, not the production one. Fix: run provider verification against a server configured identically to production, including auth middleware.

## Tradeoffs & Alternatives

### Tradeoffs

| Benefit | Cost |
|---------|------|
| Catches breaking API changes before deployment | Requires both teams to adopt Pact tooling and maintain state handlers |
| No shared environment needed for integration verification | Pact Broker is an additional infrastructure dependency |
| Independent deployability with safety guarantees | Consumer must publish updated pacts whenever client code changes |
| Fast feedback (no network, no running services) for consumer side | Provider verification requires a running (but local) service |
| Scales to many consumers without O(n²) integration environments | Can-i-deploy matrix complexity grows with number of services |

### When CDC Contract Testing Does NOT Apply

- **Third-party APIs you cannot influence** — Use recorded HTTP cassettes (nock, WireMock, Polly.js) instead. You cannot ask Stripe or Twilio to verify your Pact.
- **Same-team monolith** — Deployed and tested as a unit. Contract overhead adds ceremony without benefit.
- **GraphQL with schema stitching** — GraphQL has its own schema validation tools (GraphQL Inspector, Rover). Using Pact for GraphQL requires the Pact GraphQL plugin and adds friction vs. schema change detection.
- **Event streaming (Kafka, Kinesis) at high volume** — Message pacts verify schema but not Avro/Protobuf schema registry compatibility. Use Confluent Schema Registry's compatibility checks for Avro schemas.

### Alternatives

| Alternative | When to prefer it |
|-------------|------------------|
| **OpenAPI schema validation** (Spectral, openapi-diff) | Provider-driven teams; existing OpenAPI spec; no per-consumer granularity needed |
| **WireMock / nock stubs** | Testing against third-party APIs; one-way dependency; no need for provider verification |
| **Integration tests in shared environment** | Small team; simple topology; no independent deployment requirement |
| **Bi-directional contract testing** (PactFlow BDC) | Provider publishes OpenAPI; consumers test against it; no consumer-side mock needed |

### Known Adoption Costs

- **Tooling ramp-up:** 1–2 weeks for a team new to Pact (consumer side easier; provider state handlers take longer).
- **Pact Broker maintenance:** managed (PactFlow) vs self-hosted adds ops burden. Self-hosted requires PostgreSQL and version upgrades.
- **Organisational alignment:** CDC testing requires both the consumer team and provider team to coordinate state handlers, versioning conventions, and broker configuration. Without buy-in from the provider team, the model breaks.
- **Legacy API retrofitting:** adding contracts to existing APIs requires mapping all consumers and their actual usage patterns, which can take months for large systems.
